Task Forge v2 是一个基于 LLM 的多步骤编排系统。输入是一份真实开发场景中的 Python 脚本（dev/code_source/dirty_work/*.py），输出是一道经过 4 个独立 critic 对抗审查、并由确定性规则函数完成最终分级的 Agent 基准评测任务。当前模型层默认接 Mimo，但项目的主张不在某个模型本身，而在 workflow scaffolding、critic 审查与 deterministic calibration——一道任务是否被采纳为高质量 anchor，由一组明确的硬性条件决定，模型不参与最终判定。

术语统一：anchor（主锚任务，质量最高）／ supporting（辅助任务，可用但有上限）／ reject（拒绝，不进入评测集）。下文及分级标签均使用此三个英文 key。

01 · 项目概述

它解决的是 Agent 评测里"题目从哪来"的工程问题。

目前 Agent 评测圈的主流做法是用 LLM-as-judge 给 Agent 输出打分。这套做法在评测的可复现性上存在结构性问题：同一份输入，不同模型版本、不同温度、不同 prompt 会给出不同分数；同时优质的"评测任务"本身就是稀缺资源，需要人工产出。 Task Forge v2 的差异点是把最终分级移出模型层——LLM 只负责生成、审查与修订，任务最终是 anchor 还是 reject 由一条纯规则函数说了算，相同输入永远产出相同分级。这就让"自动产出评测任务"和"评测任务的质量保证"这两件事可以同时成立。

它不是聊天机器人，也不是 RAG 应用（虽然内部有检索层）。下面三张卡片是项目的输入、中间层与输出。

输入 · INPUT

一份真实开发场景中的 Python 脚本，带真实的依赖、函数、文件路径与副作用。来源是开发者日常工作里产出的"脏脚本"——不是教科书例子，有真实工程上下文。

中间层 · PIPELINE

13 节点 LangGraph 状态机：源代码挖掘 → 历史任务检索 → 初判 → 草稿 → 4 critic 并行审查 → 修订合并 → 终稿 → 规范化 → 规则层评分 → 落盘。低价值来源走 6 节点快速路径。

输出 · OUTPUT

一道结构化的评测任务（JSON），含任务边界、规划粒度、可观察答案；以及一份 anchor / supporting / reject 三选一的最终分级与对应 reason codes。回归脚本可直接读取该分级。

02 · 系统架构

三层架构：模型、记忆、规则各司其职。

模型层负责生成与评判，记忆层负责跨轮次复用历史产物，规则层负责最终判定。各层通过结构化 schema 通信，避免自然语言层面的歧义传播。

第一层
模型层（Model）
一个 LLM 客户端被 8 个 prompt 复用：triage、draft、4 个 critic、synthesize_review、curate。每个 prompt 都绑定对应的 Pydantic schema，模型必须按 schema 输出，否则进入 JSON 自修复流程。模型本身不掌握最终分级逻辑，每一步只回答当前节点的问题。

第二层
记忆层（Memory）
基于 LlamaIndex VectorStoreIndex，索引内容是系统在过去运行中自己产出的 artifact：源代码摘要、已审核任务、anchor 评估、critic 审查记录。检索采用三路并行（源代码主键、任务族邻居、任务形态邻居），随后按 family / shape 标签做后过滤。索引按 corpus signature 持久化在 dev/.cache/retrieval_index/，相同语料不会重复构建索引。

04 · 评分规则

最终分级由确定性判定树决定。

06 · 证据层

现在已经有一层可审计的证据，不只是流程图。

这一层证据现在来自四部分：只读 artifact audit、runtime telemetry、critic ablation、以及一个最小 `haiku vs qwen` validity probe。它们一起决定现在能说什么、还不能说什么。

对应文档：dev/report/task_forge_artifact_audit.md、dev/report/task_forge_ablation_report.md、dev/report/task_forge_validity_probe.md 与 dev/report/task_forge_reuse_inventory.md。

01

当前 artifact audit

`dev/artifacts` 现有 `33` 个 artifact 目录，其中 `17` 个是完整 assessed runs。最终桶位分布是 `4 anchor / 4 supporting / 9 reject`；评审路径分布是 `10 full review / 7 fast path`。这已经足够证明三类最终桶与早拒绝路径都真实出现过。

02

runtime telemetry 与 ablation

当前 `4-source x 2-variant` ablation 已完成：`full_review` 与 `no_critics` 都得到 `2 anchor / 1 supporting / 1 reject`，但成本差异明显。平均 `llm_call_count` 从 `6.25` 降到 `2.5`，平均时长从 `43.63s` 降到 `25.01s`。这说明 critic 路径现在的价值更像 trace richness，而不是已经证明会改变最终桶位。

03

主 eval 项目的可复用资产

主项目已经有完整的 canonical surfaces：`144` 个 Chat cells、`96` 个 CLI cells、`72` 个 Task 2 handoff cells，以及配套的 manual rubric、judge disagreement、source-run audit 与 failure taxonomy。Task Forge 可以直接复用这套 audit 结构和 `hard anchor / soft support / counterexample` 的叙事方式。

04

当前 validity read

更大的 `12-task` validity sweep 已完成，比较模型仍是 `haiku` vs `qwen`。当前方向性结论是：`reject` 桶比较成立，但 `anchor > supporting` 仍是 mixed evidence，尤其 `compress.py` 这条 archive-cleanup 线出现明显漂移。Task Forge 已据此新增 `runtime_verification_underdefined` 规则，并把新 rerun 的 `compress.py` 从 anchor 压回 supporting。也就是说，这套系统现在不只会出题，还会根据外部 validity 证据收紧自己的 deterministic rubric。

07 · 工程亮点

项目体现的 Agent 工程实践。

下列六个设计选择构成了项目作为 Agent 工程作品的核心价值。

01

多步骤编排

采用 13 节点 LangGraph 状态机：triage → draft → 4 critics 并行 → synthesize → curate → assess。每个节点的输入输出都已 schema 化，可独立单测，也可单独重放。

02

多 critic 对抗式审查

4 个 critic 分别覆盖一类失败模式（边界、泄露、范围、评测可观测性），由 synthesize_review 节点合并为统一的修订方案。每个 critic 的偏好可通过替换其 prompt 单独调整。

03

持久化记忆层

每轮运行都消费上一轮的 artifact。LlamaIndex 持久化索引按语料签名缓存——相同语料不会重复嵌入。任务本身无需感知历史，但系统会自动避免与历史任务重叠。

04

确定性评分

项目的核心论点：anchor 资格并非由 LLM 投票决定，而是由 assess_task 纯函数依据多个硬性条件判定。相同输入永远产生相同输出，这是评测工程可复现性的基础要求。

05

预算感知的路由策略

并非每份源代码都值得 8 次模型调用。初判后通过 add_conditional_edges 分流：低价值来源走 6 步快速路径，仅消耗一次模型调用即可完成。这是评测工具中常被忽视的成本意识体现。

06

结构化输出规范

每个节点的 LLM 输出均由 Pydantic 模型定型，失败时进入 JSON 自修复，再次失败才抛出异常。状态合并使用 Annotated[..., operator.add]，避免字符串拼接。失败信息能精确定位到具体 schema 字段，便于排查与回归。
